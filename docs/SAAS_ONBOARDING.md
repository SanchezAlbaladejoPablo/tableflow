# docs/SAAS_ONBOARDING.md — Per-Restaurant Onboarding Guide

_This document covers the v2.0 SaaS onboarding flow for new restaurant owners._

---

## Overview

TableFlow SaaS allows any restaurant to be onboarded in under 5 minutes via a self-service wizard. No technical knowledge required.

**Onboarding steps:**
1. Sign up → create restaurant_admin account
2. Configure restaurant details (name, timezone, operating hours)
3. Set up tables (number, capacity, area, floor plan position)
4. (Optional) Configure booking widget
5. Invite staff members

---

## Step 1 — Sign Up

Navigate to the TableFlow app URL. Click **"Register your restaurant"**.

Enter:
- Your name
- Email address
- Password (min 8 characters)
- Restaurant name

On submit:
- A `users` record is created with `role = "restaurant_admin"`
- A `restaurants` record is created and linked to the user
- A `restaurant_settings` record is created with sensible defaults
- You are logged in automatically

---

## Step 2 — Restaurant Settings

After sign-up, the onboarding wizard opens to **Step 1: Restaurant Details**.

| Field | Default | Description |
|---|---|---|
| Restaurant name | (from registration) | Displayed in the dashboard header |
| Timezone | Europe/Madrid | Used to display reservation times correctly |
| Opening time | 13:00 | Earliest allowed reservation time |
| Closing time | 23:30 | Latest allowed reservation time |
| Reservation duration | 90 minutes | Default duration per reservation slot |
| Gap between reservations | 3 hours | Minimum gap between two reservations on the same table |
| Primary color | #6366f1 | Brand color for the dashboard |
| Logo URL | (empty) | URL of your restaurant logo (shown in header) |

Click **"Save & Continue"**.

---

## Step 3 — Add Tables

The wizard shows an empty floor plan. Use the **"+ Add Table"** button to add tables.

For each table, set:
- **Table number** (e.g., 1, 2, A1, Terraza-1)
- **Capacity** (number of seats)
- **Area** (Indoor, Terraza, Bar)
- **Shape** (rectangle or circle)

After adding tables, drag them to position them on the floor plan canvas.

Click **"Save & Continue"**.

---

## Step 4 — Booking Widget (Optional)

Enable the public booking widget to allow guests to book directly from your website.

1. Toggle **"Enable booking widget"** ON
2. A widget token is generated automatically
3. Copy the embed code:

```html
<iframe
  src="https://your-tableflow-url/widget.html?token=YOUR_WIDGET_TOKEN"
  width="400"
  height="600"
  frameborder="0">
</iframe>
```

4. Paste the embed code into your restaurant website

Guests can now:
- Check availability for a given date/party size
- Submit a reservation (status = `pending`)
- Receive a confirmation email

Staff must confirm pending widget reservations in the dashboard.

---

## Step 5 — Invite Staff

From the **Settings** page, go to **Team Members**.

Click **"Invite Staff"** and enter their email address.

They will receive an email with a sign-up link. When they register, their account is linked to your restaurant with `role = "staff"`.

Staff can:
- View and manage reservations
- Update table status
- Create/edit/delete reservations

Staff **cannot**:
- Change restaurant settings
- Add/remove tables
- View or manage billing

## Default Configuration Values

The `restaurant_settings` record created at sign-up has these defaults:

```json
{
  "timezone": "Europe/Madrid",
  "default_duration_minutes": 90,
  "min_gap_minutes": 180,
  "opening_time": "13:00",
  "closing_time": "23:30",
  "booking_widget_enabled": false,
  "booking_widget_token": "<auto-generated UUID>",
  "primary_color": "#6366f1",
  "logo_url": ""
}
```

All defaults can be changed in **Settings → Restaurant Configuration**.

## PocketBase Collection Rules (Technical Reference)

For the `restaurant_admin` role, collection rules are configured as follows:

### `reservations`
```
List:   @request.auth.id != "" && @request.auth.restaurant_id = restaurant_id
View:   @request.auth.id != "" && @request.auth.restaurant_id = restaurant_id
Create: @request.auth.id != "" && @request.body.restaurant_id = @request.auth.restaurant_id
Update: @request.auth.id != "" && @request.auth.restaurant_id = restaurant_id
Delete: @request.auth.role = "restaurant_admin" && @request.auth.restaurant_id = restaurant_id
```

### `tables`
```
List:   @request.auth.id != "" && @request.auth.restaurant_id = restaurant_id
View:   @request.auth.id != "" && @request.auth.restaurant_id = restaurant_id
Create: @request.auth.role = "restaurant_admin" && @request.body.restaurant_id = @request.auth.restaurant_id
Update: @request.auth.role = "restaurant_admin" && @request.auth.restaurant_id = restaurant_id
Delete: @request.auth.role = "restaurant_admin" && @request.auth.restaurant_id = restaurant_id
```

### `restaurant_settings`
```
List:   @request.auth.id != "" && @request.auth.restaurant_id = id
View:   @request.auth.id != "" && @request.auth.restaurant_id = id
Update: @request.auth.role = "restaurant_admin" && @request.auth.restaurant_id = id
```

### Public booking widget (`reservations` — widget token scope)
```
Create: @request.auth.type = "widget" && @request.body.restaurant_id = @request.auth.restaurant_id
        && @request.body.status = "pending"
```

---

## Multi-tenant Data Isolation Guarantee

Every API request made by the frontend includes a JWT token. PocketBase evaluates the collection rule expression **before** running the query. The expression `@request.auth.restaurant_id = restaurant_id` ensures that:

1. A user from Restaurant A can never list or view Restaurant B's reservations, tables, or customers.
2. A user from Restaurant A can never create data tagged with Restaurant B's ID.
3. No client-side manipulation of the `restaurant_id` field can bypass this — the rule is enforced server-side.

This makes cross-tenant data access impossible without breaking into the PocketBase server itself.
